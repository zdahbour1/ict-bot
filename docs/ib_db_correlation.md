# IB ↔ DB Correlation via `orderRef`

**Status:** design, pending approval (no implementation yet)
**Motivation:** today's IB→DB linkage is `trades.ib_perm_id` reverse
lookup. Every caller that has an IB order and needs its DB row has
to `SELECT FROM trades WHERE ib_perm_id=X OR ib_tp_perm_id=X OR
ib_sl_perm_id=X`. That query is everywhere — reconcile PASS 2,
orphan detector, the audit trail UI. It works but it's fragile:

- `permId` is only set after IB accepts the order (not available
  at placement time in some error paths).
- Reverse lookup by `permId` across 3 columns is indirect; reading
  a single order doesn't tell you "which trade is this?" without
  a DB query.
- When IB rejects and we never get a `permId`, the order still
  appeared in TWS briefly; there's no way to tie that transient
  to a DB row at all.

The proposal: tag every order with a **client-generated correlation
ID** that lives with the order forever in IB's `orderRef` field.
The same ID goes into a new column `trades.client_trade_id`. Any
caller with an IB order can read its `orderRef` and get the
correlation ID without a DB query. Any caller with a DB row can
pull `client_trade_id` and find the matching IB orders by filtering
on `orderRef`.

---

## 1. What IB provides

### `Order.orderRef` (string)

Free-form client string, max 40 chars in practice (IB doesn't
document a hard limit but we'll cap ourselves). Passed to IB on
`placeOrder`, returned on every `orderStatus`, `execDetails`, and
`openOrder` callback, and visible in TWS under column "Order Ref".

Examples of how other systems use it:
- Strategy tags: `"momentum-v2"`, `"scalper-1m"`
- Client correlation IDs: UUIDs, sequence numbers, trade IDs

### `Order.permId` (int)

IB-assigned, globally unique across an account, never reused.
Available **only after** IB accepts the order. We already store it
in `trades.ib_perm_id`.

### `Order.orderId` (int)

Per-client sequence. NOT globally unique. Can collide across stale
sessions (today's bug: `cancel_order_by_id(3913)` could cancel
two different orders on two different clients).

### What's missing in `ib_async`'s `openOrder` snapshot

Surprisingly, `orderRef` IS included in the ib_async `Order`
object. Our current code just ignores it. It's already one field
read away in every callback — we simply haven't used it yet.

## 2. Two correlation-ID schemes (pick one)

### Option A — pre-allocated DB row id (tightly coupled)

Flow:

```sql
SELECT nextval('trades_id_seq') INTO :next_id;   -- pre-allocate PK
```

```python
next_id = get_next_trade_id()
orderRef = f"db:{next_id}"
# Place parent + TP + SL all with orderRef=f"db:{next_id}"
# After placement succeeds:
insert_trade(id=next_id, ...)    # use the pre-allocated PK explicitly
```

**Pros:**
- orderRef IS the db_id. Zero indirection. `SELECT * FROM trades
  WHERE id = <orderRef parsed>`. The simplest possible lookup.

**Cons:**
- Requires changing `trades.id` from `generated always as identity`
  to `generated by default as identity` or equivalent (so we can
  insert explicit values). Minor schema churn.
- If placement succeeds but `insert_trade` fails, we've "burned" a
  DB id that has no row (gap in sequence). Not a correctness
  problem but cosmetically annoying.
- Entry flow must coordinate sequence allocation with the IB
  placement thread — extra synchronization.

### Option B — separate correlation UUID (loosely coupled) ← RECOMMENDED

Flow:

```python
import uuid
correlation_id = uuid.uuid4().hex[:12]    # e.g. "a7c4e29b1f03"
orderRef = f"bot:{correlation_id}"
# Place parent + TP + SL all with orderRef=f"bot:{correlation_id}"
insert_trade(client_trade_id=correlation_id, ...)
# db_id assigned normally by PG; client_trade_id is a separate column
```

DDL:
```sql
ALTER TABLE trades
  ADD COLUMN client_trade_id VARCHAR(16) UNIQUE;
CREATE INDEX idx_trades_client_trade_id ON trades(client_trade_id);
```

**Pros:**
- Generated BEFORE any DB coordination — works even if DB is
  unavailable at entry time (retry later with same ID).
- Decoupled: `trades.id` stays a normal PK, `client_trade_id` is
  a pure correlation tag. Two independent concerns.
- Idempotency: if insert_trade fails mid-placement, a retry with
  the SAME `correlation_id` is safe (unique constraint catches
  a double-insert).
- Compatible with future cross-process / cross-machine placement
  (e.g. if the scanner and the IB connection are ever split).
- Works for adoption: reconcile PASS 2 reads `orderRef` from an
  IB orphan; if it starts with `bot:`, look up by
  `client_trade_id` — if found, we know this was OUR order from a
  crashed session; if not, it's truly external and we create a
  new row.

**Cons:**
- One extra column. Minor.
- One extra lookup step (`orderRef` string → parse UUID →
  `SELECT ... WHERE client_trade_id`). Still O(1) via the index.

**Verdict: Option B.** The decoupling is worth it for the failure-
mode robustness. Every downstream use case works either way; the
upside is in the error paths.

## 3. Proposed orderRef format

```
bot:<12-char-hex>           e.g. "bot:a7c4e29b1f03"
```

- **Prefix `bot:`** — reserved so we know at a glance this is ours,
  not a manual TWS order or a different strategy. Manual TWS
  orders have empty or user-typed `orderRef`.
- **12 hex chars** — ~48 bits of entropy, enough to never collide
  across a lifetime of trades; fits easily in orderRef's
  de-facto 40-char limit with plenty of room for future
  extension (e.g. `bot:a7c4e29b1f03:retry1`).
- All three orders (parent BUY, TP LMT, SL STP) carry the SAME
  orderRef — that's the defining property of "this bracket belongs
  to one logical trade."

Future extensions (not v1):
- `bot:<id>:tp` / `bot:<id>:sl` — if we need to distinguish legs
  without looking at order type.
- `bot:<id>:restore` — when reconcile PASS 4 replaces brackets
  via `place_protection_brackets`, tag them so we can tell
  "original" from "restored" in audits.

## 4. Where it plugs in

### 4.1 `broker/ib_client.py::place_bracket_order`

New kwarg `order_ref: str | None = None`. If provided, set on
parent + both children before `placeOrder`. Default None means
"don't tag" (backwards compatible).

```python
def place_bracket_order(self, option_symbol, contracts, action,
                        tp_price, sl_price, order_ref: str | None = None):
    ...
    if order_ref:
        parent.orderRef = order_ref
        tp_order.orderRef = order_ref
        sl_order.orderRef = order_ref
```

Same treatment for `place_protection_brackets`, which should get
a distinct prefix (e.g. `bot:<id>:restore`) so `bracket_restored`
audits can be identified from the order alone.

### 4.2 `strategy/trade_entry_manager.py::enter`

At the TOP of `enter`, before any gates:

```python
import uuid
correlation_id = uuid.uuid4().hex[:12]
order_ref = f"bot:{correlation_id}"
```

Threaded through `_place_order_with_timeout → select_and_enter →
place_bracket_order`. After successful placement:

```python
trade["client_trade_id"] = correlation_id
self.exit_manager.add_trade(trade)   # persists client_trade_id
```

### 4.3 `db/writer.py::insert_trade`

Accept + persist `client_trade_id`. Already straightforward once
the column exists.

### 4.4 `broker/ib_client.py` — expose `orderRef` in query results

Update `_ib_get_all_working_orders`, `_ib_find_orders_for_contract`,
`_ib_get_positions_raw` to include `order_ref` in their output
dicts. Zero cost; it's one more attribute read per iteration.

### 4.5 Consumers

**Reconcile PASS 2 (adopt):**
```python
# Before: always creates a new DB row for an IB orphan
# After:
ref = pos.get("order_ref", "")
if ref.startswith("bot:"):
    correlation_id = ref.removeprefix("bot:").split(":")[0]
    existing = session.query(Trade).filter_by(
        client_trade_id=correlation_id
    ).first()
    if existing:
        # We've seen this trade before — probably a prior session's
        # crash. Update its ib_perm_ids if needed, don't double-adopt.
        _revive_existing_trade(existing, pos)
        continue
# Fall through to normal adoption (creates new row)
```

**Orphan detector:**
```python
# Before filtering, skip orders where orderRef matches an ACTIVE
# trade's client_trade_id — belt-and-suspenders on top of the
# existing open_db_con_ids check.
```

**Audit trail UI:**
- Trades tab Audit modal could show `client_trade_id` so the user
  can cross-reference with TWS's Order Ref column.
- `/api/trades/{id}/audit` already returns all fields; just expose
  the new column.

## 5. Migration strategy

Zero-downtime:

1. Ship the DDL: column nullable, indexed.
2. Ship `insert_trade` + `place_bracket_order` changes — NEW trades
   get tagged.
3. Old trades have `client_trade_id = NULL` forever. Consumers
   handle NULL by falling back to permId lookup (today's behavior).
4. Once the column is populated for all open trades (i.e. all
   current open trades close and new ones take their place),
   consumers could optionally drop the permId fallback. No rush.

No backfill needed. Old orders in TWS don't get orderRef
retroactively (IB doesn't support modifying orderRef on existing
orders), and old trades in DB stay NULL. That's fine — the column
is for going-forward linkage.

## 6. What this enables

### Immediate
- **Simpler orphan detection.** Today: conId + DB lookup. With
  orderRef: the order tells you directly whether it's OURS and
  which trade.
- **Better reconcile adopt.** Today: crashed session's orders
  become new DB rows (orphans we then close). With orderRef: we
  resurrect the original DB row.
- **Clearer TWS inspection.** Operator opens TWS, sees `bot:a7c4e29b1f03`
  on an order, can grep the DB or audit log and immediately know
  which trade it's part of.

### Future
- **FOP extension.** When we add futures options
  (`docs/fop_live_trading_design.md`), the same correlation works
  seamlessly — `orderRef` is not contract-type specific.
- **Cross-strategy tagging.** ORB trades could get `orb:<id>`,
  VWAP `vwap:<id>`, ICT `ict:<id>`. Instant visual separation
  in TWS.
- **Tracking retries / compensations.** `bot:<id>:restore` vs
  `bot:<id>:retry1` distinguishes the original bracket from the
  compensating one placed by reconcile PASS 4.

---

## 7. What this does NOT solve

- **Order modification.** IB doesn't let you change `orderRef` on
  an existing order via `placeOrder`-with-same-ID modification
  (or at least not reliably across versions). So all 3 bracket
  orders must be tagged at creation time, not patched afterwards.
- **Historical link-up.** Orders from today's afternoon cascade
  don't get tagged retroactively. Clean slate only.

## 11. Cross-client cancel asymmetry — sell-first close mode

### The constraint
IB server-side rule: `cancelOrder(orderId)` only works from the
client that originally placed the order. Any other client gets
**Error 10147** ("OrderId only valid from owner"). This is not
a bug — it's the IB API design. There's no "as-admin" flag.

What *does* work from any client on the same account:
- `placeOrder()` — submitting new orders (including SELL)
- `reqPositions()` — reading positions (global)
- `reqAllOpenOrders()` — reading orders (not cancelling)
- `reqGlobalCancel()` — cancels EVERY order on the session
  (rejected: we plan to run multiple strategies / accounts
   and this is indiscriminate)

### Why pool-slot strict-cancel-first fails on restart
Our pool gives each slot whatever `clientId` IB hands out at
connect time. After a bot restart, the new pool slots don't
own the brackets left by yesterday's process — so
`cancel_order_by_id` → 10147 → strict-cancel-verify times out
→ `_atomic_close` refuses to send the SELL. Position stays
open with stale brackets until reconcile PASS 3/4 repairs it.

### Sell-first flow (default as of this PR)

Implemented in `strategy/exit_executor.py::_execute_exit_sell_first`,
gated by `config.CLOSE_MODE_SELL_FIRST` (default **True**):

```
1. Check IB position qty (from any client — global read).
   - qty == 0 → DB-only update, mark _bracket_fired if we
                expected brackets. No SELL.
   - qty <  0 → CRITICAL alert, abort. (Bot only does long
                options; negative qty is a naked short.)
2. Fire best-effort cancel on any live brackets — NON-BLOCKING.
   Failures (10147) are logged and ignored.
3. Send MKT SELL. (Placement works from any client.)
4. verify_brackets_cleared_post_sell(timeout=5s):
     - Poll find_open_orders_for_contract every 0.5s
     - Expect: IB auto-cancels resting TP/SL when position
       flattens (OCA + options-bracket behavior).
     - If any bracket still alive at deadline → issue
       explicit cancel and re-poll.
     - If STILL alive after explicit cancel → non-critical
       handle_error alert. Reconcile PASS 3 will clean up.
5. Re-check IB position qty (simultaneous-fill guard):
     - qty <  0 → CRITICAL alert ("naked_short_after_close").
                  Indicates bracket TP filled the same tick
                  as our MKT SELL — very rare, needs manual
                  review.
```

### Why this is safe for our tickers
- Universe is high-liquidity options (QQQ, SPY, IWM, AAPL,
  MSFT, NVDA, INTC, MU, PLTR, TSLA). MKT SELL fills instantly;
  the race window between SELL-sent and brackets-auto-cancelled
  is milliseconds, not seconds.
- Step 5 catches the simultaneous-fill corner case that the
  old strict-cancel-first prevented by fiat. Fires a CRITICAL
  alert rather than silently ignoring.
- `reqGlobalCancel` is explicitly NOT used — it would cancel
  orders belonging to other strategies running on the same
  IB account.

### Rollback
Set `CLOSE_MODE_SELL_FIRST=false` (either in settings table or
env) → legacy `cancel_all_orders_and_verify` path runs
unchanged. No code deletion — the legacy path is still fully
live.

### Tests
`tests/unit/test_close_sell_first.py` (11 tests):
- qty==0 skips SELL + flags _bracket_fired
- qty<0 aborts with critical handler
- happy path LONG → sell_call; SHORT → sell_put
- SELL fires even when pre-cancel raises 10147
- verify_brackets_cleared_post_sell: terminal quick, explicit
  cancel, handle_error on survival after explicit cancel
- best_effort_cancel_brackets: swallows errors, noop when empty
- legacy mode still reachable with CLOSE_MODE_SELL_FIRST=false

### Config
```
CLOSE_MODE_SELL_FIRST     = True   (new default)
POST_SELL_BRACKET_TIMEOUT = 5.0s   (how long to wait for IB
                                     auto-cancel before issuing
                                     explicit cancel)
```

### Future: stable clientId routing (ARCH-007, deferred)
The cleaner long-term fix is to pin pool slots to deterministic
clientIds (10, 11, 12, 13) and persist `trades.ib_client_id` at
entry. Reconcile then knows exactly which slot owns each order
and can route cancels without fan-out or sell-first fallback.
Lower priority now that sell-first unblocks Thursday testing.
